Task RF-1 — Critical: Block delivered invoice deletion

Problem:
Delivered invoices can still be deleted/moved to recycle bin instead of going through the cancel flow. This bypasses the intended stock-reversal/audit path.

- server/routes/invoices.ts
- client/src/components/invoices/InvoiceActionsDropdown.tsx
- relevant invoice tests

- DELETE /api/invoices/:id must reject invoices where status is "delivered" OR stockDeducted is true.
- Return a clear 400/409 message telling the user to use Cancel Invoice instead.
- The UI must not show "Delete" for delivered invoices. It should show/use the existing Cancel Invoice flow.
- Draft/submitted invoices may keep the existing delete-to-recycle behavior.

Tests:
- Delivered invoice DELETE returns error and invoice remains.
- Delivered invoice action menu does not offer Delete.
- Draft/submitted invoice delete still works as before.
- Cancel invoice flow still works.

Avoid:
- Do not change stock movement logic.
- Do not change PDF/print/export.
- Do not change invoice totals/VAT logic.


Task RF-2 — High/Medium: Make master-data recycle-bin deletes atomic

Problem:
Brand, customer, and supplier delete routes insert a recycle-bin row before the actual delete. If the delete fails due to foreign-key/history usage, the recycle bin can contain a fake deleted record while the live record still exists.

Affected files:
- server/routes/brands.ts
- server/routes/customers.ts
- server/routes/suppliers.ts
- optionally server/routes/products.ts only to verify it already behaves safely

Required behavior:
- Recycle-bin insert and actual delete/deactivate must happen in one transaction.
- If deletion fails, no recycle-bin row should remain.
- FK/history failures should return a clear 400/409, not create partial state.
- Keep existing user-facing behavior otherwise.

Tests:
- Try deleting a supplier used by a PO: route fails cleanly and no recycle-bin row is created.
- Same idea for customer/brand if referenced.
- Deleting an unused record still creates recycle-bin row and removes/deactivates the live record.

Avoid:
- Do not redesign recycle bin.
- Do not change restore behavior except as needed by the transaction.

Problem:
PO create/update checks that items is a non-empty array, but the totals helper can silently skip invalid rows. This can create/update a PO with zero valid line items. PO POST is also not fully atomic.

Affected files:
- server/routes/purchase-orders.ts
- server/lib/purchaseOrderTotals.ts
- relevant PO tests

Required behavior:
- After computePurchaseOrderTotals(), reject with 400 if computed.items.length === 0.
- Apply this to POST and PUT-with-items.
- Wrap PO POST header creation, item inserts, totals update, and company snapshot update in one transaction.
- Keep PUT transaction behavior.
- Tighten numeric parsing:
  - reject malformed numeric strings like "12abc"
  - reject negative/non-positive FX rates when provided
  - reject malformed productId where an item row was intentionally supplied
- Keep accepting normal UI payloads.

Tests:
- POST with items array but zero valid computed items returns 400 and creates no PO.
- PUT with invalid/empty computed items returns 400 and preserves existing PO lines.
- POST failure during item insert does not leave a header-only PO.
- Normal PO create/update still works.

Avoid:
- No schema changes.
- Do not touch GRN logic except tests if needed.
- Do not touch PO PDF/print/export layout.
